# 전산회계 문제은행 — 설계 문서 (Design Spec)

- 작성일: 2026-07-21
- 상태: 승인됨 (구현 계획 대기)
- 코드네임: `acct_quiz`

## 1. 한 줄 요약

전산회계 1급·2급, 세무 2급 기출을 **토큰 0으로 파싱**해 4지선다 연습 툴로 제공하는 웹앱.
차별점은 콘텐츠 텍스트가 아니라 **기능**(실무분개 4지선다 채점 · 유형별 필터 · 오답노트)이며,
수익은 유형별 정리 페이지의 **애드센스**(목표 하루 약 1,000원)로 낸다.

## 2. 목표 / 비목표

**목표**
- 기출 문제를 4지선다로 풀고 자동 채점.
- 유형별(가지급금·부가세·감가상각 등)로 문제 모아보기.
- 틀린 문제 오답노트(로그인 유저).
- 유형별 정리 페이지를 구글에 색인시켜 애드센스 유입 확보.

**비목표 (지금 안 함)**
- KcLep 화면 통짜 흉내(전자세금계산서·전표메뉴 UI 재현). 과임.
- 자유입력 계정과목 채점. 전부 4지선다로 통일하므로 불필요.
- **LLM 콘텐츠 생성 일절 금지 — 토큰 0이 하드 제약.** 문제/해설을 AI로 재작성하지 않는다.

## 3. 핵심 제약 (하드 룰)

1. **토큰 0**: 문제·해설 생성에 LLM 사용 금지. 오직 추출·규칙분류·발행. (at_blog 컴활 파이프라인과 동일 원칙)
2. **comcbt 클론 금지**: 기출 텍스트를 그대로 나열해 SEO를 노리지 않는다. 개별 문제 텍스트는 comcbt가 원본(canonical)이라 신규 복사본은 구글 중복 필터에서 짐.
3. **오리지널리티는 기능·구조에서**: 유형별 재구성 + 인터랙티브 채점 + 오답노트. 애드센스는 "콘텐츠 글"이 아니라 "기능성 퀴즈 툴"로 통과를 노린다.

## 4. 정직한 리스크

- 개별 문제 텍스트는 comcbt와 겹침 → 그 페이지들은 구글이 약하게 평가할 수 있음. 그래서 **색인·수익 동선은 유형별 정리 페이지**(comcbt에 없는 조합)와 툴 기능에 둔다.
- 하루 1,000원 = 월 약 3만원. 한국 교육 애드센스 RPM 대략 2~4천원/1000뷰 → **하루 300~500 페이지뷰** 필요 → 유형 정리 페이지 수십~수백 개 색인 필요. 토큰0 대량 파싱으로 페이지 볼륨을 확보해 이 조건을 채운다.
- 저작권: 전산세무회계 기출은 한국세무사회(민간) 저작물. comcbt가 15년간 기출+광고로 운영 = CBT 형식엔 실무상 강한 대응 없음(회색지대). 리스크 낮게 보되, 원문 통짜 나열은 피하고 유형 재구성/툴 형태로 노출.

## 5. 기술 스택

- **Next.js (App Router)** — SSG/SSR로 유형 페이지 색인 + 애드센스.
- **Supabase** — Postgres + Auth(구글 OAuth) + RLS.
- **Vercel** 무료 배포.
- 세 서비스 무료 티어로 하루 천원 규모 커버.

대안 검토: 순수 정적(Astro)은 더 가볍지만 오답노트/로그인에 Supabase가 필요하므로 Next로 통일.

## 6. 데이터 모델

핵심 테이블 2개.

```sql
-- 문제: 4개 기능 전부 여기서 파생
questions (
  id            uuid pk,
  subject       text,   -- 전산회계1급 | 전산회계2급 | 세무2급
  category      text,   -- 이론 | 실무분개
  type_tag      text,   -- 가지급금 | 부가세 | 감가상각 ...
  stem          text,   -- 문제 본문
  choices       jsonb,  -- ["보기1","보기2","보기3","보기4"]
  answer_idx    int,    -- 0~3
  explanation   text,   -- 기출 파싱 시 함께 추출(없으면 null)
  source        text,   -- 참고 회차 (예: 전산회계1급 120회)
  created_at    timestamptz default now()
)

-- 풀이 기록: 오답노트 재료
attempts (
  id           uuid pk,
  user_id      uuid references auth.users,
  question_id  uuid references questions,
  chosen_idx   int,
  is_correct   bool,
  created_at   timestamptz default now()
)
```

기능↔쿼리 매핑:
- 실무분개 / 이론 → `category` 필터
- 유형별 → `type_tag` 필터
- 오답정리 → `attempts where is_correct = false` (user 기준)

RLS: `attempts`는 `user_id = auth.uid()`인 행만 read/write. `questions`는 공개 read.

## 7. 페이지 (= 수익 동선)

| 경로 | 렌더 | 로그인 | 역할 |
|---|---|---|---|
| `/[subject]/[type_tag]` | SSG | 불필요 | **유형별 정리. 구글 유입 + 애드센스. 돈줄.** |
| `/quiz` | CSR/SSR | 익명 허용 | 4지선다 CBT, 자동 채점, 익명은 localStorage 임시저장 |
| `/wrong` | SSR | **필요** | 오답노트 |
| `/` | SSG | 불필요 | 과목·유형 인덱스(색인 허브) |

원칙: **오답노트만 로그인.** 문제풀이·정리페이지는 익명 허용(유입 안 막게).

## 8. 콘텐츠 파이프라인 (토큰 0)

at_blog 컴활 수집 파이프라인 재사용.

1. **추출**: comcbt / 기출 PDF에서 문제·보기·정답·(있으면)해설 파싱. LLM 없음.
2. **분류**: `type_tag`를 키워드 규칙으로 부여. 예) 본문에 "가지급금" 포함 → `type_tag=가지급금`. 규칙 테이블은 코드에 상수로.
3. **적재**: Supabase `questions` upsert. 중복(같은 stem) 스킵.

분류 규칙은 정확도보다 커버리지 우선. 미매칭 문제는 `type_tag=미분류`로 넣고 나중에 규칙 추가.

## 9. 단계 (마일스톤)

1. **M1 스키마 + 시드**: Supabase 테이블·RLS + 한 과목(전산회계1급) 유형 5개 소량 적재.
2. **M2 풀이+정리**: `/quiz` 4지선다 채점 + `/[subject]/[type_tag]` SSG 정리 페이지.
3. **M3 로그인+오답**: 구글 OAuth + `/wrong` 오답노트.
4. **M4 수익화**: 애드센스 신청 → 파싱 확대로 유형 페이지 볼륨 늘리기.

## 10. 검증 포인트

- 파싱: 채점 정답(`answer_idx`)이 실제 정답과 일치하는지 회차 샘플로 대조.
- 채점 로직: 선택 인덱스 == answer_idx 판정 단위 테스트 하나.
- RLS: 남의 오답노트 못 읽는지 확인.
- SEO: 유형 페이지가 색인·유니크한지(중복 아님) 배포 후 서치콘솔 확인.
